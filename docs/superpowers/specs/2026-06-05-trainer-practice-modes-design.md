# CubeAssist ŌĆö Trainer Practice Modes Design Spec

**Date:** 2026-06-05
**Status:** Approved (brainstorming complete)
**Reference spec:** `docs/superpowers/specs/2026-05-14-cubeassist-design.md`

## Goal

Replace the Trainer placeholder with three working practice modes:

1. **F2L** ŌĆö drill recognition + execution of all 41 first-two-layers cases.
2. **Cross** ŌĆö drill cross solving at a chosen difficulty (2ŌĆō8 movers), with a color-neutral option.
3. **Look-ahead** ŌĆö a metronome / constant-pace drill that forces no-pause solving.

All three assume the user solves on a **physical cube** and uses an on-screen
WCA-style timer. No interactive virtual cube. Simple 2D SVG diagrams show the
case / target where it helps recognition.

This builds the supporting infrastructure the original spec deferred to "Plan 3"
(scramble generation, WCA timer, cube diagrams) but scopes it to exactly what
these three modes need.

## Decisions Locked During Brainstorming

- **Physical practice + on-screen timer.** No virtual cube simulator.
- **Simple 2D diagrams now** (not text-only). Built from a facelet cube model.
- **Look-ahead = metronome / constant-pace only.** No slow-solve or cross+1 modes.
- **F2L = all 41 cases**, with a correctness test for every case.
- **Cross = difficulty selector (2ŌĆō8 movers)**, scramble + timer only, no shown
  solution. A cross solver is built solely to *grade* scrambles by difficulty.

## Architecture

### Decision 1 ŌĆö Small custom cube engine, no library

A ~150-line facelet model (54 stickers; the 6 face turns as fixed permutation
arrays) plus alg parsing and inversion. No `cubing.js` / `cubejs` dependency ŌĆö
matches the project's lightweight, pure-testable, no-backend ethos. This single
engine powers F2L diagrams, scramble application, and the cross solver.

### Decision 2 ŌĆö F2L content is derived from the algorithm

For each of the 41 cases we author only `{ name, group, algorithm, recognition }`.
Everything else is derived by the cube engine:

- **setup scramble** = `invert(algorithm)`
- **case diagram** = `applyAlg(solved(), setup)`

Because the diagram is generated by applying the *inverse of the solving alg* to
a solved cube, the diagram is **always consistent with the algorithm** ŌĆö there is
no separate hand-authored diagram that could drift out of sync. A standard F2L
alg only disturbs its slot + the top layer, so its inverse-setup leaves the cross
and the other three slots solved, which is exactly the F2L trainer premise (set
up from a solved cube, solve one pair).

The front-right (FR) slot is the canonical slot for all 41 cases.

### Decision 3 ŌĆö Cross difficulty via a BFS distance table

The cross is the 4 D-layer edges. Modeled as a coordinate over (position,
orientation) of those 4 edges (~255k reachable states). A single BFS from the
solved-cross state builds a distance table giving the **exact optimal solve
length (0ŌĆō8)** for any state. `optimalCrossLength(state, color)` is then a table
lookup. `generateCrossScramble(targetLen, color)` generates random scrambles
until one whose cross length equals `targetLen`.

## Routes

The original spec's `/trainer/:caseId` scheme doesn't map cleanly onto these
three modes, so the Trainer is restructured:

| Route | View |
|---|---|
| `/trainer` | Landing ŌĆö three mode cards (F2L ┬Ę Cross ┬Ę Look-ahead) |
| `/trainer/f2l` | F2L trainer |
| `/trainer/cross` | Cross trainer |
| `/trainer/lookahead` | Look-ahead (metronome) trainer |

`App.tsx` already mounts `/trainer/*`, so only nested routing inside the Trainer
area changes.

## Trainer: F2L (`/trainer/f2l`)

**Mode toggle:**
- **Single case** ŌĆö pick one of the 41 cases from a list; drill it repeatedly.
- **Mixed** ŌĆö a random case each rep; the case name/alg is **hidden until after
  the solve** (recognition focus). The diagram is still shown (you recognize from
  the cube, the diagram confirms after).

**Per-rep flow:**
1. Show the **setup scramble** (WCA notation). User applies it to a solved cube;
   only the FR pair is now in the top layer.
2. Show the **2D oblique case diagram** ŌĆö U face (3├Ś3) + the top row of the F and
   R faces. This is the conventional way F2L cases are drawn.
3. User solves the pair; WCA timer records the time.
4. After the solve: reveal case **name**, **algorithm**, and **recognition** note
   (in Mixed mode this is the reveal moment).
5. Running **Ao5 / Ao12** shown in-session.

## Trainer: Cross (`/trainer/cross`)

**Controls:**
- **Difficulty** ŌĆö segmented selector `2 ┬Ę 3 ┬Ę 4 ┬Ę 5 ┬Ę 6 ┬Ę 7 ┬Ę 8` movers (optimal
  cross length).
- **Color** ŌĆö Fixed (white) or **Random (color-neutral practice)**. In random
  mode each scramble picks a random cross color and the bottom-face diagram shows
  which color to solve.

**Per-rep flow:**
1. `generateCrossScramble(difficulty, color)` produces a scramble whose optimal
   cross length equals the chosen difficulty.
2. Show scramble + a small **bottom-face diagram** indicating the target cross
   color. No solution is shown.
3. User solves only the cross; WCA timer records the time.
4. Running Ao5 / Ao12 shown in-session.

## Trainer: Look-ahead (`/trainer/lookahead`)

A constant-pace metronome drill ŌĆö the canonical look-ahead trainer.

**Controls:**
- **Tempo** ŌĆö turns-per-second (TPS) slider, e.g. 1.0ŌĆō5.0.
- **Start / stop** metronome.
- Optional **timer** running alongside.

**Per-rep flow:**
1. Show a **full-solve scramble** (`generateScramble(20)`).
2. User starts the metronome and solves keeping **every turn on the beat with no
   pauses**. Audio click (WebAudio) + a visual beat pulse.
3. Optional WCA timer for total solve time.

The point is cadence, not a specific case, so there is no per-case diagram here.

### Cube engine sketch (`lib/cube.ts`)

```ts
// 54 facelets, index 0..53, grouped 9-per-face in order U, R, F, D, L, B.
export type Facelets = number[];            // each entry = face color id 0..5
export function solved(): Facelets;
export function parseAlg(s: string): Move[]; // "R U R'" -> moves
export function invert(s: string): string;   // "R U R'" -> "R U' R'"
export function applyAlg(state: Facelets, alg: string): Facelets;
```

Moves are the 18 standard `{U,D,L,R,F,B} ├Ś {'', "'", "2"}`. Each quarter turn is
a fixed facelet permutation; `2` and `'` derive from it. Wide/slice/rotations are
**not** required for F2L algs at the FR slot but `parseAlg` should tolerate `y`
rotations and wide moves if any authored alg needs them (fall back: include the
small set actually used by the 41 algs).

### Cross model sketch (`lib/cross.ts`)

```ts
// The 4 cross edges for a given bottom color. State = their (slot, flip).
export function optimalCrossLength(scramble: string, color: Color): number; // 0..8
```

Built once via BFS over the cross-edge coordinate from the solved-cross state;
distances cap at 8 (white cross is always Ōēż8 moves). A module-level memoized
table avoids recomputing per call.

## Data: `f2l.json` enrichment

Existing shape (consumed by `Profile.tsx`):

```json
{ "id": "f2l-01", "name": "F2L 01" }
```

Enriched shape (additive ŌĆö `Profile.tsx` ignores the new fields, no change there):

```json
{
  "id": "f2l-01",
  "name": "F2L 01 ŌĆö corner on top, edge on top (basic insert)",
  "group": "Basic / connected pair",
  "algorithm": "R U' R'",
  "recognition": "White corner faces front, edge sits beside it on top."
}
```

`name` becomes descriptive (the Profile checklist gets nicer labels for free).
`group` buckets cases (e.g. "Pair joined on top", "Corner in slot", "Edge in
slot", "Pieces separated") for the Single-case picker.

## Persistence

Each completed session appends lightweight `DrillRecord`s to the existing
`profile.drillHistory` (schema unchanged) so data accrues for the future
Dashboard recommendation engine:

- F2L Single case ŌåÆ `caseId` = the F2L case id.
- F2L Mixed ŌåÆ `caseId` = `"f2l-mixed"`.
- Cross ŌåÆ `caseId` = `"cross"`.
- Look-ahead ŌåÆ `caseId` = `"lookahead"`.

No cross-session history *viewer* is built in this pass ŌĆö that is the Dashboard's
job in a later plan. In-session, only live Ao5/Ao12 is shown.

## Testing Strategy

**Unit (Vitest):**

- `cube.ts`
  - `solved()` has 9 of each color.
  - `R` applied 4├Ś = identity; sexy move `R U R' U'` applied 6├Ś = identity.
  - `invert("R U R'")` === `"R U' R'"`; `applyAlg(applyAlg(s, a), invert(a))` === `s`.
- `cross.ts`
  - `optimalCrossLength` of the solved state = 0.
  - A scramble of a single quarter turn that displaces one cross edge Ōēź 1.
  - The BFS table covers all reachable states with max distance Ōēż 8.
- `scramble.ts`
  - `generateScramble(n)` returns `n` moves, never two consecutive same-face,
    never three consecutive same-axis.
  - `generateCrossScramble(k, color)` returns a scramble with
    `optimalCrossLength === k` (loop until match; bounded retries).
- **F2L data correctness** ŌĆö for **every** of the 41 cases: starting from
  `solved()`, applying `setup = invert(algorithm)` then `algorithm` returns to
  `solved()`, AND the setup alone leaves the cross + the three non-FR slots
  solved (only the FR slot + U layer are disturbed). This is the key guard that
  the authored algs are clean FR-slot F2L algs.

**Component (React Testing Library):**

- `Timer` ŌĆö spacebar holdŌåÆreadyŌåÆstartŌåÆstop produces a positive time; respects an
  inspection-off setting.
- `TrainerCross` ŌĆö selecting a difficulty renders a scramble (cross-length
  assertion may be mocked for speed).
- `TrainerF2L` ŌĆö Mixed mode hides the case name until after a solve.

## Out of Scope (this pass)

- Interactive / virtual cube.
- OLL / PLL / full-solve trainers (existing spec's other Trainer modes).
- Cross **solution** display or an OLL/PLL solver.
- Cross-session drill-history viewer / charts (Dashboard's job).
- Slow-solve and cross+1 look-ahead variants.
- Animated move-by-move walkthroughs (Library's job).
